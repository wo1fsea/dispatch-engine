# Event Protocol

Use this reference when changing Dispatch Engine run-state or event-log behavior.

## Run State Shape

```text
.dispatch/
  plans/
    <plan-id>.json
  runs/
    <run-id>/
      run.json
      events.jsonl
      decisions.jsonl
      workstreams/
        01-implementation.json
      artifacts/
      reviews/
      validation/
```

`<run-id>` is a UTC timestamp-like identifier generated by the bundled runtime.
The latest run is the lexicographically last directory under `.dispatch/runs/`.

`.dispatch/plans/` is the expected location for Dispatch Engine-generated explicit plan files before import. If an operator supplies a plan from outside `.dispatch/`, the runtime may record the source, but generated non-project runtime content still belongs under `.dispatch/`.

`run.json` is the run summary and compatibility surface. It includes:

- `kind`: currently `plan`
- `run_id`
- `repo_root`
- `state_dir`
- `objective`
- `status`: currently `planned` for dry-run plans
- `created_at`
- `updated_at`
- `workstreams`: summary objects for each planned workstream
- `decisions`: pending decision objects, or an empty list
- `plan`: imported plan metadata when available, including `plan_id`, `schema_version`, `source_path`, `created_by`, `created_at`, and `target_repo`
- `repo_context`: Codex-prepared repository context summary from the imported plan when available
- `review`: review strategy from the imported plan when available

Each `workstreams/<id>.json` file is the durable state for a single planned
workstream. It includes:

- `id`
- `title`
- `scope`
- `files`
- `depends_on`
- `status`
- `validation`
- `created_at`
- `updated_at`

`decisions.jsonl` is append-only and exists even when no decision has been
created. Pending decision entries include:

- `id`
- `question`
- `reason`
- `status`
- `created_at`
- `updated_at`

`artifacts/` is reserved for run-scoped generated artifacts. `reviews/` is reserved for reviewer reports and acceptance records. `validation/` is reserved for runtime-captured validation commands, outputs, and summaries.

Do not commit `.dispatch/` state.

## Event Shape

```json
{
  "ts": "2026-05-01T12:00:00Z",
  "type": "workstream.planned",
  "actor": "dispatch-engine",
  "workstream": "01-implementation",
  "payload": {
    "title": "Implement objective"
  }
}
```

Events are append-only. Correct mistakes with new events instead of rewriting history unless the run is explicitly discarded.

Required fields:

- `ts`: UTC ISO-8601 timestamp ending in `Z`
- `type`: event type
- `actor`: event writer, usually `dispatch-engine`
- `payload`: event-specific object, empty when no data is needed

Optional fields:

- `workstream`: workstream id when an event applies to a specific workstream

## Initial Event Types

`run.created`

- Written once after run state is initialized.
- Payload includes `run_id` and `objective`.

`plan.imported`

- Written once after an explicit dispatch plan is validated and imported into run state.
- Payload includes `plan_id`, source path or descriptor, and workstream count.

`workstream.planned`

- Written once for each planned workstream.
- Includes `workstream`.
- Payload includes `title`.

`workstream.scheduled`

- Written when the future scheduler marks a workstream ready for a worker.
- Includes `workstream`.
- Payload may include dependency and parallel group context.

`worker.completed`

- Written when a worker returns a report for a workstream.
- Includes `workstream`.
- Payload includes worker status and report location under `.dispatch/runs/<run-id>/artifacts/` or another run-scoped runtime directory.

`review.completed`

- Written when a reviewer accepts, rejects, or requests changes for a worker result.
- Includes `workstream`.
- Payload includes review status and report location under `.dispatch/runs/<run-id>/reviews/`.

`validation.completed`

- Written when validation evidence is recorded for a run or workstream.
- May include `workstream`.
- Payload includes validation status and report location under `.dispatch/runs/<run-id>/validation/`.

`decision.created`

- Written once for each pending decision created during planning.
- Payload includes `decision_id` and `question`.

`plan.created` is superseded by `run.created`. Treat it as historical only
unless a compatibility fixture explicitly requires it.

## CLI Readers

- `python3 scripts/de.py status <repo>` reads the latest run summary.
- `python3 scripts/de.py status <repo> --run-id <run-id>` reads a selected run.
- `python3 scripts/de.py tail <repo>` prints events from the latest run.
- `python3 scripts/de.py tail <repo> --run-id <run-id>` prints events from a selected run.

Both `status` and `tail` support `--json`.
