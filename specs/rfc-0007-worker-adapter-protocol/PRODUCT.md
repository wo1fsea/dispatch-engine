# Worker Adapter Protocol Product Spec

## Summary

Dispatch Engine can now launch and supervise a coordinator-only provider CLI process. The next baseline defines how that coordinator makes implementation work observable without doing the implementation itself.

`rfc-0007-worker-adapter-protocol` introduces the first worker adapter protocol: worker registration, workstream assignment, prompt snapshotting, stdout/stderr log capture, report shape, lifecycle events, status visibility, and conservative protocol violations. The goal is not full scheduling yet; the goal is a durable contract the coordinator can use to declare "this worker is doing this scoped work, here is its output, here is the evidence."

## Goals / Non-goals

- Goal: Define a worker adapter protocol for registered implementation agents.
- Goal: Keep coordinator-only boundaries intact; coordinators may dispatch workers but must not directly implement project-file changes.
- Goal: Add a centralized worker prompt template under `references/prompts/`.
- Goal: Store worker prompt snapshots, logs, reports, and heartbeat streams under `.dispatch/runs/<run-id>/`.
- Goal: Require worker reports before workstream implementation output is accepted.
- Goal: Expose worker counts and assignments through `de status`.
- Goal: Emit worker lifecycle events: `agent.spawned`, `workstream.assigned`, `agent.completed`, `agent.failed`, and `protocol.violation`.
- Goal: Detect obvious protocol violations such as implemented workstreams without registered implementation agents or missing worker reports.
- Non-goal: Implement automatic dependency scheduling across multiple workstreams.
- Non-goal: Implement safe parallel execution.
- Non-goal: Implement reviewer acceptance or validation execution as complete end-to-end gates.
- Non-goal: Attribute arbitrary filesystem diffs to a process with perfect certainty.
- Non-goal: Require real Codex or Claude worker binaries in unit tests.

## Behavior Invariants

1. A worker, reviewer, or validator must be registered before its output is treated as valid implementation evidence.
2. Worker registration records live under `.dispatch/runs/<run-id>/agents/<agent-id>.json`.
3. Worker prompt snapshots live under `.dispatch/runs/<run-id>/prompts/<agent-id>.md`.
4. Worker stdout/stderr logs live under `.dispatch/runs/<run-id>/logs/<agent-id>.stdout.log` and `.stderr.log`.
5. Worker reports live under `.dispatch/runs/<run-id>/reports/<agent-id>.json`.
6. Worker heartbeat streams live under `.dispatch/runs/<run-id>/heartbeats/<agent-id>.jsonl`.
7. A worker record includes role, provider, profile, status, workstream, assigned files, allowed write roots, prompt path, report path, log paths, and timestamps.
8. A worker report includes status, summary, changed files, validation evidence, questions/blockers, and residual risks.
9. `de status` reports agent counts by role/status and active workstream assignments.
10. A workstream marked implemented/completed without a registered implementation agent report is a protocol violation.
11. A worker report that declares changed files outside assigned scope is a protocol violation.
12. The coordinator prompt continues to instruct that implementation belongs to registered workers, reviewers, or validators.
13. Unit tests use fake worker provider executables or direct protocol helpers, not real provider CLIs.

## States and Edge Cases

- Workstream has no assigned worker.
- Worker registered but not started.
- Worker running with heartbeat.
- Worker completed with report.
- Worker failed with reason.
- Worker blocked with questions.
- Worker report is malformed JSON.
- Worker report omits required fields.
- Worker report names changed files outside assigned scope.
- Workstream is marked implemented/completed but no implementation-agent report exists.
- Coordinator attempts to register a worker with no workstream.
- Two workers claim the same workstream.
- Worker prompt snapshot already exists.
- Legacy run lacks `prompts/`, `reports/`, `logs/`, or `heartbeats/`.

## Open Questions

- Should the first worker adapter launch real provider processes, or should rfc-0007 stop at protocol helpers and fake-provider tests?
- Should worker IDs be caller-provided, generated by the runtime, or both?
- Should changed-file scope checks be exact path matching, prefix matching, or repo-native glob matching in the MVP?
- Should worker reports be written only by adapters, or may a coordinator write a report on behalf of an external worker process?
