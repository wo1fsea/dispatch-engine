# Coordinator Spawn Contract Product Spec

## Summary

Dispatch Engine should not take over subagent scheduling from the provider coordinator. The coordinator is the main agent for deciding when to spawn workers, reviewers, or validators, whether work should be serial or parallel, and how to react to blocked work. Dispatch Engine's role is to make those coordinator-spawned agents observable, queryable, resumable, and reviewable through `.dispatch/` state.

`rfc-0008-coordinator-spawn-contract` defines the contract for coordinator-spawned agents: the coordinator may spawn subagents using provider-native mechanisms, but it must register every spawned agent, record prompt snapshots, maintain status/heartbeat state, collect durable reports, and emit lifecycle events before any output is accepted as implementation evidence.

## Goals / Non-goals

- Goal: Clarify that the coordinator owns subagent spawning decisions.
- Goal: Keep Dispatch Engine out of provider-specific subagent orchestration internals for this baseline.
- Goal: Require every coordinator-spawned worker, reviewer, or validator to be registered in `.dispatch/runs/<run-id>/agents/`.
- Goal: Require prompt snapshots, logs, heartbeats, reports, and lifecycle events for spawned agents.
- Goal: Let external interactive Codex query active subagent counts and statuses through `de status` and `de tail`.
- Goal: Preserve the coordinator-only rule: coordinator may spawn and supervise, but not directly implement project-file changes.
- Goal: Document how coordinator-spawned agents should use the rfc-0007 worker report protocol.
- Non-goal: Implement a deterministic scheduler.
- Non-goal: Implement a DE-owned worker launch adapter.
- Non-goal: Implement provider-specific subagent APIs for Codex, Claude, or other tools.
- Non-goal: Guarantee process-level attribution of every file edit.

## Behavior Invariants

1. The coordinator decides whether to spawn subagents.
2. Dispatch Engine records subagent state; it does not need to know provider-specific spawn internals.
3. A spawned worker/reviewer/validator must be registered before work assignment is treated as valid.
4. A spawned agent must have a prompt snapshot path under `.dispatch/runs/<run-id>/prompts/`.
5. A spawned agent must have stdout/stderr/log paths under `.dispatch/runs/<run-id>/logs/` when available.
6. A spawned agent must keep heartbeat or status updates while active.
7. A spawned agent must write durable role-specific evidence under `.dispatch/runs/<run-id>/reports/`, `reviews/`, or `validation/`.
8. `de status` reports active, blocked, completed, and failed spawned agents from `.dispatch/`.
9. Workstream implementation evidence requires a valid registered implementation-agent report.
10. Coordinator-written project changes remain protocol violations unless explicitly approved by a recorded decision.

## Coordinator Responsibilities

When spawning a worker:

1. Choose the workstream and scope from imported run state.
2. Register the worker in `agents/<agent-id>.json`.
3. Render and write the worker prompt snapshot.
4. Emit `agent.spawned` and `workstream.assigned`.
5. Start the provider-native subagent however the coordinator sees fit.
6. Keep heartbeat/status state current.
7. Ensure the worker writes a durable report.
8. Mark the worker completed, blocked, or failed.
9. Avoid accepting worker output if report validation or scope checks fail.

The same pattern applies to reviewers and validators once their report protocols exist.

## Open Questions

- Should rfc-0008 add CLI helper commands for coordinator use, or should file/API helpers remain enough?
- Should spawned agent IDs be generated by runtime helpers, coordinator-supplied, or both?
- Should `de status` distinguish "spawned by coordinator" from "registered by external operator"?
- Should heartbeats be required on a time interval before background execution exists?
